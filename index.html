Defining Tooling for SugarCRM Developers

Greg Levine & Jon Whitcraft, SugarCRM Engineers
You won't find anyone who believes they have reached peak productivity while developing on the SugarCRM platform.
It's not Rails.
But we want you to be more productive.
And us too.
So, to get there, we need to know more about who we are and what we need.
The following are attempts to describe how various SugarCRM developers work and what things may cause them to fall short of being productive.

SugarCRM Engineer

my workflow
I clone the sugarcrm/Mango repo. If I haven't done that already.
I check out a branch. If I'm not already on the branch I want to use.
I make some changes.
But I'm working in a pre-compiled codebase.
So I have to use a file-watcher program to continuously run changes through the build process, which writes files to a web server. This is our compilation process.
I manually test my changes in my browser.
I chdir to my source code on the web server to run tests.
I switch back to my repo to make more changes.
I use a step debugger* to debug the source code on the web server. Now I have two instances of my IDE running. Yuck! *We use Xdebug.
I do some more switching between source code directories while I debug and make changes.
I commit my changes. And often push to GitHub and open a pull request.
my problems
All of the the switching between environments. Alas, as long as there is a build step, there must be some element of switching between source code environments.
I must support a large matrix of stacks and configurations.
I have to remember how to configure things differently to build out each of those stacks locally.
I can't fully replicate any stack because my local web server is on OS X, which is not even a supported OS for production.
So I have to learn other tools and configurations if I want to build to other environments for full replication.
I rely on a remote CI process to tell me when there are issues that may break the build.
There is no canonical documentation on how to perform certain actions, like building a new SugarObject or creating a certain type of extension.
And the documentation that does exist may not be up to date.
So I must learn everything I can about what I'm doing before I even start working.
The prescribed method for creating a new module is to build it in Studio and copy the results back into the repo.
Everytime I checkout a branch, I have to be aware that the database may not work and I should just rebuild or reinstall before I head down any rabbit holes.

Partner Developer

Partners that strictly perform client-work, partners that provide pre-configured solutions, OEM partners, SugarCRM Professional Services, and SugarCRM Enterprise Professional Services are considered in this use case.
my workflow
I clone a repo and/or checkout a branch*. If I haven't done that already. *Possibly not, but this is a sample workflow.
I make some changes in a compiled codebase. Welp, we already have one major difference in workflows. sigh
I manually test my changes in my browser.
I realize I need to run Quick Repair & Rebuild. That major difference just came full cicle. It's a different flavor of build step.
I manually test my changes in my browser again.
I commit my changes.
Now I have to work on something for a different client.
So I clone a different repo.
Rinse and repeat.
my problems
My machine does not fully replicate the stack(s) that my company uses.
I don't really have any canonical tests that I can run or I had to build them myself.
I may not have access to SugarCRM's test suite, which means I may not know if I broke anything. This problem is multi-dimensional and is not necessarily solvable with a single tool alone. But it's a problem related to my workflow nonetheless.
I may need to maintain repos for different clients or for different verticals, which requires me to context switch and rebuild my database often. Among other issues that might crop up due to context switching.
Without a canonical way of performing certain actions, it is hard to guarantee that my changes are upgrade safe. I can use Studio but that seems strange when I want to build a reusable package that is not tied to a particular instance.

Client Developer

Standalone clients and SugarCRM iApps are considered in this use case.
my workflow
I clone a repo and/or checkout a branch*. If I haven't done that already. *Possibly not, but this is a sample workflow.
I make some changes in a compiled codebase.
I manually test my changes in my browser.
I realize I need to run Quick Repair & Rebuild.
I manually test my changes in my browser again.
I commit my changes.
Rinse and repeat.
my problems
A mix of subsets of the problems experienced by SugarCRM Engineers and Partner Developers.

Holy smokes, Batman! It's an endless series of bombs!

But maybe all is not hopeless.
The goal of this discussion is to better understand what you, the SugarCRM developer, needs and how we can help.
Maybe we failed to represent your use case or misrepresented it.
We want to know.
What gives you a headache?

SugarCRM Development Kit

The following are purely conceptual. We're just riffin'.
command-line tool
A command-line tool becomes the client interface to a development kit that solves as many of your needs as possible. All tasks are performed based on the current working directory.
generators
Studio exists so that administrators can make changes in a GUI, without having to understand code.
But you are a developer; you don't want to drop into a GUI all the time.
And all you want to do is stub out a new module or insert a new value into a dropdown.
There are many things that could be built around the concept of code generators. Some are trivial and some are super complex. You shouldn't have to consult documentation every time you want to perform some boilerplate action.
package management
Want to download and install packages from Sugar Exchange?
Want to download and install packages from other sources?
linter
How about a tool that can perform static code analysis on your code according to SugarCRM's internal standards and help you to identify issues that may cause integration or upgrade problems?
app management
The development kit ships with a Vagrantfile that defines all of the stacks on which you can run a SugarCRM instance.
You issue a command with arguments defining the stack configuration you want to use and a virtual machine is provisioned.
Your source code is shared with the virtual machine so that any changes are immediately available.
You can hit the app in your browser.
Under the covers, the tool takes care of certain responsibilities like, rebuilding the database, so you don't have to.
You can issue other commands to run tests.
You can drop in plugins to add new commands that make you even more productive.
Maybe you can provision the same instance on multiple stacks so that you can run tests against the entire supported stack fleet. Assuming your machine can handle the load.
Instead of provisioning machines locally, maybe you want to provision them on AWS. It shouldn't matter to us.
performance testing
Run performance testing tools against your SugarCRM development instance.
profiling
Profile your SugarCRM development instance. More or less this would handle any configuration or setup.
plugins
Write your own plugins to expand on what the development kit offers you. For instance, maybe you want it to perform some role in your deployment strategy.
The idea here would be that you could fork the development kit* and customize it to your liking. *And contribute back upstream.
If you needed to change some configurations in the Vagrantfile, then have it.
Just want to add a new command, but don't want it to be lost when you pull from upstream?
No problem.
